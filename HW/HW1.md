# SASD2025-HW-HW1.md
Martin Fowler พูดว่าซอฟต์แวร์ไม่ใช่แค่เขียนโค้ดให้รันได้ แต่โครงสร้างของมันมีผลกับทุกอย่างทั้งความเร็วในการพัฒนา การแก้บั๊ก การเพิ่มฟีเจอร์ และค่าใช้จ่ายระยะยาว
หลายทีมมัก มองข้ามเรื่องโครงสร้างซอฟต์แวร์ เพราะคิดว่าเดี๋ยวค่อยแก้ทีหลัง แต่พอระบบใหญ่ขึ้น จะกลับมาแก้ยาก แพง และช้าเขาย้ำว่า การตัดสินใจด้านซอฟต์แวร์ตั้งแต่แรก
เช่น แยกระบบยังไง ใช้เทคโนโลยีแบบไหน มีผลยาวๆ ต่อธุรกิจ ไม่ใช่แค่เรื่องเทคนิค

ภาพนี้บอกอะไร ( นาทีที่ 11:00 )
แกนล่าง (time) = เวลาที่ผ่านไป
แกนตั้ง (Cumulative Functionality) = ฟีเจอร์หรือความสามารถของระบบที่เพิ่มขึ้นเรื่อยๆ
<ul>
  ความหมายของรูป
  <li>ช่วงแรกถ้าไม่ออกแบบอะไรเลย จะเพิ่มฟีเจอร์ได้เร็วมาก เขียนโค้ดและได้ฟีเจอร์นั้นแสดงออกมาเลยทำให้ดูเหมือนคืบหน้าเร็วแต่พอเวลาผ่านไประบบเริ่มยุ่ง โค้ดพันกัน แก้ยาก การเพิ่มฟีเจอร์ใหม่ทำได้ช้าลงเรื่อยๆ</li>
</ul>

เส้นกำแพงตรงกลาง กำแพงนี้คือเส้นแบ่งว่าอะไรที่ผู้ใช้เห็น กับ อะไรที่ซ่อนอยู่ข้างในระบบ ( นาทีที่ 10:00 )
<ul>
  External Quality 
  <li>Pleasant User Interface: หน้าตาใช้ง่าย ดูดี ใช้แล้วไม่งง</li>
  <li>Few Defects: บั๊กน้อย ระบบไม่ค้าง ไม่พังบ่อย</li>
</ul>

<ul>
  Internal Quality
  <li>Good Modular Design: โค้ดเป็นระเบียบ แยกส่วนดี แก้ไขง่าย ขยายง่าย</li>
</ul>

<ul>
  สิ่งที่จะเกิดถ้า
  <li>ภายในแย่ -> แก้ยาก -> เพิ่มฟีเจอร์ช้า -> บั๊กเพิ่ม -> ผู้ใช้เริ่มรู้สึก</li>
  <li>ภายในดี -> พัฒนาได้ต่อเนื่อง -> ระบบเสถียร -> ผู้ใช้พอใจ</li>
</ul>
